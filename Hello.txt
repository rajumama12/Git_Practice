Topic 4 — Your First Commit
First understand — the 3 stages of Git
Before we type any command, understand this diagram:
Working Directory  →  Staging Area  →  Repository
  (your files)         (git add)       (git commit)

  "I edited files"   "I selected      "I permanently
                      which ones        saved them"
                      to save"
Think of it like this 🏦:
Git StageBanking AnalogyWorking DirectoryYou filled a withdrawal form (made changes)Staging AreaForm is submitted but not yet processedRepository (Commit)Transaction is posted to the ledger permanently

Step 1 — Create a file
bashecho "Hello Git! This is my first file." > hello.txt
Check it's there:
bashls
Output:
hello.txt

Step 2 — Check status
bashgit status
Output:
On branch master

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        hello.txt

nothing added to commit but untracked files present
Git is saying → "I see a new file but I'm not tracking it yet!"

Step 3 — Add file to Staging Area
bashgit add hello.txt
Now check status again:
bashgit status
Output:
On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   hello.txt
File is now in the staging area — ready to be committed ✅

Step 4 — Commit the file
bashgit commit -m "Add hello.txt - my first commit"
Output:
[master (root-commit) a1b2c3d] Add hello.txt - my first commit
 1 file changed, 1 insertion(+)
 create mode 100644 hello.txt
You just saved your first version permanently! 🎉
The -m flag is your commit message — always write something meaningful so you remember what you changed and why.

Step 5 — Check the log
bashgit log
Output:
commit a1b2c3d4e5f6... (HEAD -> master)
Author: Doon1 <memescreator009@gmail.com>
Date:   Sat Jun 28 10:35:00 2025

    Add hello.txt - my first commit
This is your history log — every commit ever made. Like Oracle's redo log! 😄

Topic 6 — Undoing Mistakes
This is one of the most important topics in Git. Everyone makes mistakes — Git lets you undo them safely.
There are 3 different undo situations. We'll cover them one by one.

Situation 1 — You edited a file but haven't staged it yet
Let's create this situation:
bashecho "Oops! This was a mistake." >> Hello.txt
git status
Output:
modified:   Hello.txt
You realise — I don't want this change at all!
Undo it:
bashgit restore Hello.txt
Now check:
bashgit status
nothing to commit, working tree clean
The mistake is gone. File is back to how it was after your last commit. ✅

⚠️ Warning: git restore permanently throws away your uncommitted changes. Use carefully — there's no recycle bin here!


Situation 2 — You staged a file but haven't committed yet
bashecho "Another mistake." >> Hello.txt
git add Hello.txt
git status
Output:
Changes to be committed:
        modified:   Hello.txt
You realise — I don't want to commit this!
Undo staging (unstage):
bashgit restore --staged Hello.txt
Check status:
bashgit status
Changes not staged for commit:
        modified:   Hello.txt
The file is unstaged — back to working directory. The change still exists in the file, but it's no longer queued for commit.
Then if you want to discard the change too:
bashgit restore Hello.txt

Situation 3 — You committed but want to undo the commit
bashecho "Bad commit incoming." >> Hello.txt
git add Hello.txt
git commit -m "This was a bad commit"
git log --oneline
Output:
9f3a1b2 (HEAD -> master) This was a bad commit
32e0485 add how to modify and then commit changes doc
f07718d Add hello.txt - my first commit
You realise — I want to undo that last commit!
Undo last commit (keep changes in working directory):
bashgit reset HEAD~1
Check log:
bashgit log --oneline
32e0485 (HEAD -> master) add how to modify and then commit changes doc
f07718d Add hello.txt - my first commit
Bad commit is gone! ✅ But your file changes are still there — just uncommitted. You can fix and recommit properly.

HEAD~1 means "one commit before HEAD" — like saying "go back 1 step."
